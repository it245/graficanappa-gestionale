2. Descrizione del Sistema

Il sistema MES di Grafica Nappa srl è un'applicazione web sviluppata su misura per gestire l'intero ciclo produttivo dello stabilimento tipografico, dalla ricezione degli ordini fino alla spedizione del prodotto finito. Il sistema sostituisce la precedente gestione manuale (fogli Excel, comunicazioni verbali) con un flusso digitalizzato, tracciabile e misurabile.

Scopo del sistema

Obiettivi principali:
  • Centralizzare la gestione degli ordini di produzione provenienti dall'ERP Onda
  • Fornire a ogni operatore una dashboard dedicata con le fasi di propria competenza
  • Tracciare tempi di lavorazione, pause, quantità prodotte e scarti per ogni fase
  • Calcolare automaticamente le priorità di produzione in base a scadenze e tempi stimati
  • Monitorare in tempo reale lo stato delle macchine da stampa (OEE tramite Prinect)
  • Gestire le lavorazioni esterne affidate a fornitori terzi
  • Verificare il completamento di tutte le fasi prima della spedizione
  • Analizzare costi di produzione, margini e performance per la direzione
  • Generare report direzionali multi-foglio per il controllo di gestione

Processi gestiti

Il sistema copre i seguenti processi operativi, descritti in ordine cronologico di flusso produttivo:

# Processo Descrizione
1 Import ordini Ricezione automatica o manuale degli ordini dall'ERP Onda o da file Excel. Creazione delle fasi di lavorazione con mapping automatico ai reparti.
2 Scheduling produzione Calcolo automatico delle priorità basato su data di consegna, tempi stimati e sequenza delle fasi. Il responsabile può intervenire manualmente.
3 Avanzamento fasi Ogni operatore avvia, mette in pausa (con motivo) e termina le fasi di propria competenza. Il sistema traccia tempi, operatori e quantità.
4 Lavorazioni esterne Gestione delle fasi affidate a fornitori esterni, con selezione fornitore, tracciamento stato e tempistiche.
5 Spedizione Verifica completamento di tutte le fasi dell'intera commessa, conferma spedizione con aggiornamento automatico di tutti gli stati.
6 Monitoraggio macchine Integrazione real-time con Prinect per l'XL106: cicli buoni/scarto, tempi, operatori, calcolo OEE.
7 Analisi costi e margini Calcolo automatico del costo di produzione per reparto, confronto con valore ordine e costo materiali per determinare il margine.
8 Reportistica Report operatori, commesse, produzione, direzione con export Excel multi-foglio.

Utenti del sistema

Tipologie di utenti:
  • Operatori di produzione (~30 persone): accedono alla dashboard del proprio reparto, avviano/pausano/terminano fasi, inseriscono quantità e note
  • Responsabile produzione (owner): visione completa dello scheduling, import ordini, gestione priorità, dettaglio commesse, export Excel, report spedizione
  • Amministratore sistema (admin): gestione anagrafica operatori (CRUD), configurazione tariffe per reparto, accesso a tutti i report (produzione, direzione, costi), dashboard Prinect e Fiery

4. Database

Il database principale del sistema è un file SQLite situato in database/database.sqlite. La scelta di SQLite è motivata dalla semplicità di deploy (nessun server database da configurare), dalla natura single-server dell'applicazione e dai volumi di dati gestiti (migliaia di ordini/anno, non milioni). Il sistema accede anche in sola lettura al database SQL Server dell'ERP Onda per l'import degli ordini.

4.2 Migrazioni

Il sistema utilizza il meccanismo di migration di Laravel per gestire lo schema del database in modo versionato. Attualmente sono presenti 43+ file di migrazione nella directory database/migrations/, che coprono la creazione delle tabelle, l'aggiunta di colonne, le modifiche ai dati e le correzioni. Ogni migrazione è eseguibile con il comando php artisan migrate e garantisce l'aggiornamento incrementale dello schema senza perdita di dati.

Importante: prima di eseguire le migrazioni in produzione, effettuare sempre un backup del file database/database.sqlite. Essendo un file unico, il backup consiste nella semplice copia del file.

5. Integrazioni

Il sistema MES si integra con cinque sistemi esterni per scambiare dati in modo automatico o semi-automatico. Le integrazioni sono progettate con un principio di resilienza: il MES deve continuare a funzionare anche quando uno o più sistemi esterni non sono disponibili.

5.1 Onda ERP (SQL Server)

Parametro Valore
Tipo integrazioneLettura diretta database via connessione sqlsrv
DirezioneOnda → MES (sola lettura)
FrequenzaManuale (bottone nella dashboard owner) o comando Artisan onda:sync
ServizioApp\Services\OndaSyncService
Livello di rischioMedio
FallbackImport manuale da file Excel
Dati importati da Onda:
  • Ordini: numero commessa, cliente, descrizione articolo, quantità ordinata, data consegna, valore ordine, costo materiali
  • Fasi: elenco fasi di lavorazione per ogni ordine, con sequenza, tempi stimati (avviamento + produzione), quantità carta
  • Clienti: ragione sociale, codice cliente
Logica di sincronizzazione (OndaSyncService):
  • Legge tutti gli ordini aperti (non completati) da Onda
  • Per ogni ordine, crea o aggiorna il record nella tabella ordini
  • Per ogni fase dell'ordine, crea o aggiorna il record nella tabella ordine_fasi
  • Mappa automaticamente ogni nome fase al reparto corrispondente tramite il catalogo fasi
  • Gestisce la deduplicazione per ordini monofase, con massimo 2 fasi e multifase
  • Al termine, ricalcola automaticamente le priorità di tutti gli ordini attivi

5.2 Prinect (Heidelberg XL106)

Parametro Valore
Tipo integrazioneREST API con autenticazione Basic Auth
EndpointConfigurabile via variabile d'ambiente PRINECT_API_URL
Device monitoratoHeidelberg XL106 — ID dispositivo “4001”
DirezionePrinect → MES (sola lettura)
FrequenzaReal-time su dashboard, sync schedulabile via comando Artisan
ServizioApp\Services\PrinectSyncService
Livello di rischioBasso
Dati acquisiti da Prinect:
  • Attività macchina: commessa in lavorazione, tipo attività (avviamento, produzione, fermo)
  • Cicli: fogli buoni e fogli scarto per ogni commessa
  • Tempi: tempo di avviamento e tempo di produzione per ogni lavoro
  • Operatori: nome dell'operatore macchina per ogni attività
  • Cambi lastre: numero e durata dei cambi lastre per ogni lavoro
  • Anteprima foglio: immagine del foglio di stampa imposto (visibile in dashboard operatore e owner)
Calcolo OEE (Overall Equipment Effectiveness):
  • Disponibilità = Tempo operativo / Tempo pianificato (turno pianificato: 23 ore/giorno)
  • Performance = Cicli effettivi / Cicli teorici (velocità nominale: 18.000 cicli/ora)
  • Qualità = Fogli buoni / (Fogli buoni + Fogli scarto)
  • OEE = Disponibilità × Performance × Qualità

Il MES funziona regolarmente anche senza connessione a Prinect. I dati Prinect sono complementari e arricchiscono le informazioni di produzione, ma non sono necessari per il flusso operativo base. Quando Prinect è disponibile, i dati vengono utilizzati per il pre-fill automatico dei campi quantità prodotta e scarti al termine delle fasi di stampa offset.

6.3 Sicurezza di rete

Misure di sicurezza:
  • Rete locale: l'applicazione è accessibile esclusivamente dalla rete interna aziendale; non è esposta su Internet
  • Password bcrypt: le password degli operatori sono memorizzate con hash bcrypt (costo computazionale elevato, resistente a brute-force)
  • Sessione PHP: autenticazione basata su sessione con timeout configurabile
  • CSRF Protection: Laravel include di default la protezione CSRF su tutti i form POST
  • Validazione input: tutti gli input utente sono validati lato server con le regole Laravel
  • File .env: credenziali e chiavi API sono escluse dal repository Git e memorizzate nel file .env

7. Funzionalità Principali

Questa sezione descrive in dettaglio tutte le funzionalità operative del sistema MES, organizzate per area funzionale. Ogni funzionalità viene descritta dal punto di vista dell'utente e, dove rilevante, con dettagli tecnici sull'implementazione.

7.1 Import ordini

L'import degli ordini è il punto di ingresso di tutti i dati produttivi nel sistema MES. Sono disponibili due canali di import, entrambi con lo stesso risultato finale: creazione degli ordini e delle relative fasi di lavorazione nel database MES.

A) Import da Onda ERP (canale primario):
  • Il servizio OndaSyncService si connette al database SQL Server di Onda
  • Legge tutti gli ordini di produzione aperti (non completati, non annullati)
  • Per ogni ordine, recupera le fasi di lavorazione dalla tabella PRDDocFasi
  • Crea o aggiorna i record nelle tabelle ordini e ordine_fasi
  • Ogni nome di fase viene mappato automaticamente al reparto corrispondente tramite il catalogo fasi (100+ mapping)
  • Gestisce la deduplicazione: ordini monofase, con massimo 2 fasi, multifase
  • Al termine della sincronizzazione, ricalcola le priorità di tutti gli ordini attivi
  • Attivabile: bottone “Sync Onda” nella dashboard owner, oppure comando php artisan onda:sync
B) Import da Excel (canale fallback):
  • Upload manuale di un file Excel dalla dashboard owner
  • Le stesse regole di mapping fasi e deduplicazione si applicano
  • Utilizzabile quando la connessione Onda non è disponibile
  • Comando alternativo: php artisan import:excel-tutto {file} per import da riga di comando
Mapping fasi a reparti:

Il sistema mantiene un catalogo di oltre 100 nomi di fase, ciascuno associato al reparto di competenza. Quando un ordine viene importato, ogni fase viene automaticamente assegnata al reparto corretto. Esempi di mapping:

7.2 Dashboard Operatore

  • Il sistema identifica i reparti assegnati all'operatore dalla tabella operatore_reparto
  • Vengono mostrate solo le fasi dei reparti di competenza con stato 0 (Caricato), 1 (Pronto) o 2 (Avviato)

7.3 Dettaglio Commessa (pagina operatore)

La pagina dettaglio commessa è il centro operativo dove l'operatore gestisce la lavorazione di una specifica fase. La pagina è divisa in diverse sezioni informative e operative.

Sezione 1 — Informazioni ordine:
  • Cliente e ragione sociale
  • Descrizione articolo/lavoro
  • Quantità ordinata
  • Note prestampa (istruzioni dalla prestampa)
  • Operatore prestampa assegnato
  • Commento produzione (note del responsabile)
Sezione 2 — Anteprima foglio di stampa (Prinect):
  • Se disponibile, mostra l'immagine del foglio di stampa imposto recuperata da Prinect
  • L'immagine è cliccabile per aprirla a dimensione intera in una nuova finestra
  • Visibile per le fasi di stampa offset; non mostrata se Prinect non è disponibile
Sezione 3 — Fase selezionata con pulsanti azione:
  • Avvia: imposta lo stato a 2 (Avviato), registra data/ora inizio e operatore
  • Pausa: apre un menu per selezionare il motivo della pausa (Attesa materiale, Problema macchina, Pranzo, Altro). Lo stato della fase diventa il testo del motivo
  • Termina: apre un modal Bootstrap con:
    • Quantità target (sola lettura, valore previsto dall'ordine)
    • Quantità prodotta (obbligatorio, campo numerico)
    • Scarti (facoltativo, default 0)
    • Pre-fill automatico da Prinect per fasi STAMPA XL (fogli buoni e fogli scarto)
    • Bottone “Conferma e Termina” per salvare e impostare stato 3
  • Riprendi: disponibile quando la fase è in pausa, riporta lo stato a 2 (Avviato)
Sezione 4 — Note operatore:
  • Campo testuale editabile dove l'operatore può inserire annotazioni libere
  • Le note vengono salvate e sono visibili a tutti gli utenti che accedono alla commessa
Sezione 5 — Altre fasi del reparto:
  • Elenco delle altre fasi dello stesso reparto per la stessa commessa
  • Sola lettura, ma cliccabili per passare alla gestione di un'altra fase
Sezione 6 — Fasi di altri reparti:
  • Panoramica dello stato di avanzamento della commessa negli altri reparti
  • Sola lettura, utile per capire a che punto è il lavoro complessivo

7.4 Ciclo di vita della fase

Ogni fase di lavorazione attraversa un ciclo di vita ben definito, rappresentato da uno stato numerico (o testuale nel caso delle pause). Le transizioni sono gestite dal sistema con regole precise.

Stato Codice Descrizione Badge
Caricato 0 Fase appena importata, in attesa che le fasi precedenti vengano completate Caricato
Pronto 1 Tutte le fasi prerequisite sono completate (stato ≥ 3); la fase può essere avviata Pronto
Avviato 2 Fase in lavorazione; operatore e timestamp registrati Avviato
Terminato 3 Fase completata; quantità prodotta e scarti registrati Terminato
Spedito 4 Ordine spedito al cliente; stato finale Spedito
Transizioni automatiche:
  • 0 → 1 (Caricato → Pronto): avviene automaticamente quando tutte le fasi precedenti dell'ordine raggiungono lo stato 3 (Terminato) o superiore. Il sistema verifica la sequenza delle fasi definita nell'ordine.
  • Tutte le fasi → 4 (Spedito): quando l'operatore di spedizione conferma l'invio, il sistema imposta automaticamente lo stato 4 su tutte le fasi dell'ordine.
Gestione delle pause:
  • Quando un operatore mette in pausa una fase, lo stato diventa una stringa descrittiva del motivo
  • Motivi disponibili: Attesa materiale, Problema macchina, Pranzo, Altro
  • Viene creato un record nella tabella pausa_operatores con timestamp inizio e motivo
  • Alla ripresa, il record viene aggiornato con il timestamp fine e lo stato torna a 2 (Avviato)
  • Il tempo in pausa viene escluso dal calcolo delle ore effettive di lavorazione
Terminazione della fase (modal v1.2):
  • Il click su “Termina” apre un modal di conferma (non più un semplice dialog)
  • Quantità target: mostrata in sola lettura come riferimento (dalla quantità dell'ordine)
  • Quantità prodotta: campo obbligatorio, deve essere ≥ 0
  • Scarti: campo opzionale, default 0, deve essere ≥ 0
  • Pre-fill Prinect: per le fasi STAMPA XL, i campi vengono pre-compilati con i dati reali dal servizio PrinectSyncService (fogli buoni → qta prodotta, fogli scarto → scarti). L'operatore può modificare i valori prima di confermare
  • I valori vengono salvati nei campi qta_prod e scarti della tabella ordine_fasi

Contenuti principali:

  • Report spedizione giorno: elenco degli ordini spediti nella giornata corrente
  • Pulsanti azione:

    • Sync Onda (sincronizzazione con ERP)

7.6 Dashboard Spedizione

La dashboard spedizione è dedicata al reparto spedizione e gestisce l'ultima fase del ciclo produttivo: la verifica del completamento e l'invio dell'ordine al cliente.

Sezioni della dashboard:
  • “Da consegnare”: ordini con tutte le fasi non-spedizione completate (stato ≥ 3). Il sistema controlla l'intera commessa (tutti gli ordini collegati), non solo il singolo ordine. Un ordine appare qui solo quando ogni fase di ogni reparto (esclusa la spedizione stessa) è stata terminata.
  • “Spedite oggi”: ordini con stato 4 (Spedito) e data_fine nella giornata corrente. Conferma visiva delle spedizioni effettuate.
  • “In attesa”: ordini ancora in lavorazione, con percentuale di completamento calcolata come rapporto tra fasi terminate e fasi totali.
Lavorazioni esterne:
  • Pagina dedicata accessibile tramite bottone dal KPI box nella dashboard spedizione
  • Mostra tutte le fasi assegnate al reparto “esterno”
  • Selezione fornitore tramite dropdown con lista fornitori precaricata da Excel
  • Opzione “Altro” per fornitori non in lista
  • Pulsanti: Avvia (con fornitore), Pausa (con motivo), Termina (con modal qta/scarti)
Invio spedizione:
  • Pulsante “Invia” disponibile per gli ordini nella sezione “Da consegnare”
  • L'azione imposta automaticamente lo stato 4 (Spedito) su tutte le fasi dell'ordine
  • Registra la data/ora di spedizione
  • Invia notifica email automatica (se configurata)
  • È possibile forzare la spedizione anche se alcune fasi non sono ancora completate (con conferma esplicita)

7.7 Dashboard Prinect

La dashboard Prinect è riservata agli amministratori e fornisce un monitoraggio completo della macchina da stampa offset Heidelberg XL106, con dati acquisiti in tempo reale dall'API Prinect.

Informazioni visualizzate:
  • Status macchina: stato corrente dell'XL106 in tempo reale (in produzione, in avviamento, ferma, manutenzione)
  • Attività oggi: elenco delle commesse lavorate nella giornata, con cicli buoni/scarto, tempi di avviamento e produzione
  • Attività ultimi 7 giorni: storico della settimana per analisi trend
KPI e indicatori:
  • Cicli buoni / scarto: conteggio totale e per commessa
  • Percentuale scarto: rapporto scarti/totale, con alert se supera la soglia
  • Tempo avviamento vs produzione: rapporto tra tempo dedicato all'avviamento (setup) e alla produzione effettiva
  • OEE (Overall Equipment Effectiveness): indicatore complessivo di efficienza (Disponibilità × Performance × Qualità)
  • Produzione per giorno: grafico/tabella con cicli giornalieri
  • Produzione per operatore: confronto performance tra operatori macchina
  • Top commesse per cicli: classifica delle commesse con più cicli di stampa
  • Statistiche cambi lastre: numero e durata media dei cambi lastre
Alert automatici:
  • Macchina ferma Se la macchina risulta ferma durante l'orario di produzione
  • Scarto alto Se la percentuale di scarto supera la soglia configurata
  • OEE basso Se l'OEE scende sotto il livello target

Le attività Prinect vengono sincronizzate nel database locale (tabella prinect_attivita) tramite il comando php artisan prinect:sync-attivita per consentire analisi storiche anche quando il server Prinect non è raggiungibile.

7.8 Costi e Margini

Il modulo costi e margini permette all'amministrazione di analizzare la redditività della produzione, confrontando il valore degli ordini con i costi effettivi di lavorazione e materiali.

Configurazione tariffe orarie:
  • Ogni reparto ha una tariffa oraria configurabile (euro/ora)
  • Le tariffe hanno date di validità (inizio/fine) per gestire gli aggiornamenti nel tempo
  • La configurazione è accessibile dall'area admin
  • I dati sono memorizzati nella tabella costi_reparto
Calcolo costi di produzione:
Costo lavoro = Ore effettive × Tariffa oraria del reparto

Ore effettive = Tempo totale fase - Tempo in pausa

Costo totale = Costo lavoro + Costo materiali (da Onda)
  • Le ore effettive escludono automaticamente il tempo delle pause registrate
  • Il costo materiali proviene dai dati importati da Onda
  • Il calcolo è eseguito per singola fase, poi aggregato per ordine
Calcolo margine:
Margine (%) = (Valore ordine - Costo materiali - Costo lavoro) / Valore ordine × 100
  • Il valore ordine proviene dai dati Onda
  • Un margine positivo indica redditività; un margine negativo indica una perdita
Report costi disponibili:
  • Per periodo: settimana, mese, trimestre, anno
  • Confronto: con il periodo precedente (es. mese corrente vs mese precedente)
  • Dettaglio per reparto: ore lavorate e costo per ciascun reparto
  • Dettaglio per commessa: costo totale e margine per singola commessa
  • Export Excel: tutti i report sono esportabili in formato Excel

7.10 Priorità automatiche

Il sistema calcola automaticamente la priorità di ogni fase di lavorazione utilizzando una formula che tiene conto della data di consegna, dei tempi stimati di lavorazione e della sequenza delle fasi. La priorità determina l'ordine di visualizzazione nelle dashboard e guida gli operatori sulle lavorazioni da eseguire per prime.

Formula di calcolo priorità:

Priorità = GiorniMancanti - OreStimate / 24 + OrdineFase / 100

dove:
  GiorniMancanti = (DataConsegna - Oggi) / 86400
  OreStimate = TempoAvviamento + (QtaCarta / CopieOra)
  OrdineFase = valore da configurazione fasi_priorita (range 0-999)
Interpretazione del valore di priorità:
  • Valore negativo: la fase è urgente o già in ritardo rispetto alla data di consegna; apparirà in cima alla lista con evidenziazione
  • Valore prossimo a zero: la fase deve essere lavorata a breve per rispettare la scadenza
  • Valore positivo: la fase ha margine temporale; la consegna è nel futuro

Le priorità vengono ricalcolate automaticamente ad ogni sincronizzazione Onda e possono essere ricalcolate manualmente con il comando php artisan priorita:ricalcola. Il responsabile produzione può inoltre intervenire manualmente sulle priorità dalla pagina di scheduling.

8. Procedura di Deploy

Il deploy del sistema MES è stato progettato per essere semplice e rapido, con un tempo di fermo stimato inferiore a un minuto. Non è necessario alcun processo di build frontend poiché l'applicazione utilizza Blade templates e Bootstrap via CDN.

Passi per il deploy:
  1. Backup preventivo: copiare il file database/database.sqlite in una posizione sicura
  2. Pull del codice: eseguire git pull origin master nella directory del progetto
  3. Migrazioni database: eseguire php artisan migrate --force per applicare eventuali nuove migrazioni
  4. Verifica funzionale: accedere al sistema e verificare che le dashboard siano operative
Note importanti:
  • Non è necessario eseguire composer install per gli aggiornamenti normali (solo per il primo deploy o se cambiano le dipendenze PHP)
  • Non è necessario alcun build frontend (nessun npm run build)
  • Le modifiche a view e controller sono immediatamente attive dopo il git pull
  • Il tempo di fermo effettivo è limitato ai pochi secondi della migrazione database
  • In caso di problemi, il rollback consiste nel ripristino del file SQLite dal backup e nel git checkout della versione precedente

11. Rischi Identificati

L'analisi dei rischi è parte integrante della documentazione del sistema. Di seguito sono elencati i principali rischi identificati, con valutazione dell'impatto, della probabilità e delle contromisure previste o in fase di implementazione.

Rischio Impatto Probabilità Contromisura
Server singolo locale (no ridondanza) Alto Media In roadmap: server secondario o virtualizzazione per disaster recovery. Attualmente mitigato da backup periodici e procedura di ripristino documentata.
Database SQLite file-based (no replication nativa) Medio Bassa Backup regolari del file database.sqlite. In roadmap: backup automatico off-site. Volumi di dati compatibili con SQLite (non critici per performance).
Dipendenza da rete interna per accesso Medio Bassa La rete interna aziendale è gestita e monitorata. In caso di problemi di rete, il sistema non è raggiungibile ma non perde dati. In roadmap: valutazione accesso VPN per emergenze.

Connessione Onda non disponibile Medio Bassa Fallback tramite import Excel. Il MES continua a funzionare con i dati già importati. La sincronizzazione può essere ripresa quando Onda torna disponibile.
Aggiornamenti non testati prima del deploy Medio Media Ogni release include una lista di test consigliati (vedi relazioni tecniche). In futuro: ambiente di staging per test pre-deploy.

12. Procedura di Ripristino di Emergenza

In caso di guasto completo del server o di corruzione del sistema, la seguente procedura consente di ripristinare il MES su un nuovo server. Il tempo stimato per il ripristino completo è di 2-4 ore, a seconda della disponibilità dell'hardware e della velocità di download delle dipendenze.

Prerequisiti:
  • Server Windows funzionante con accesso alla rete aziendale
  • Accesso al repository Git (locale o remoto)
  • Backup recente del file database.sqlite
  • Copia del file .env con le configurazioni (chiavi API, credenziali)
Procedura di ripristino passo-passo:
  1. Ripristino server Windows: installare o ripristinare il sistema operativo Windows Server. Verificare la connettività di rete con gli altri sistemi aziendali (Onda, Prinect).
  2. Installazione PHP 8.x: installare PHP 8.x con le estensioni necessarie: sqlite3, pdo_sqlite, sqlsrv, pdo_sqlsrv, mbstring, openssl, curl, xml. Verificare il funzionamento con php -v.
  3. Installazione Composer: scaricare e installare Composer (gestore dipendenze PHP) se non già presente.
  4. Clone del repository: eseguire git clone del repository nella directory di destinazione. Verificare che tutti i file siano presenti.
  5. Installazione dipendenze: eseguire composer install --no-dev nella directory del progetto per installare tutte le dipendenze PHP.
  6. Ripristino database: copiare il file database.sqlite dal backup nella directory database/ del progetto. Verificare i permessi di lettura/scrittura.
  7. Configurazione ambiente: ripristinare il file .env con le configurazioni corrette:
    • Chiave API Prinect (PRINECT_API_URL, credenziali Basic Auth)
    • Connessione Onda SQL Server (ONDA_DB_HOST, ONDA_DB_DATABASE, credenziali)
    • Configurazione email SMTP
    • Path del file Excel per sincronizzazione
    • Generare una nuova APP_KEY con php artisan key:generate
  8. Migrazioni (se necessario): eseguire php artisan migrate per applicare eventuali migrazioni mancanti rispetto al backup del database.
  9. Avvio web server: avviare il server web (PHP built-in con php artisan serve --host=0.0.0.0 --port=80 oppure configurare Apache/Nginx).
  10. Test funzionale completo:
    • Accesso login operatore → verificare dashboard con fasi
    • Accesso login owner → verificare dashboard e import
    • Test sync Onda → verificare connessione SQL Server
    • Test dashboard Prinect → verificare connessione API
    • Test termina fase → verificare modal e salvataggio dati
    • Test spedizione → verificare cambio stato e notifica email
Tempo stimato per il ripristino:
FaseTempo stimato
Setup server + PHP30-60 minuti
Clone + composer install15-30 minuti
Ripristino database + .env15-30 minuti
Test funzionale30-60 minuti
Totale2-4 ore
Raccomandazione critica: verificare periodicamente (almeno mensilmente) che il backup del file database.sqlite sia recente e integro. Verificare che una copia del file .env sia conservata in una posizione sicura e separata dal server principale. Senza questi due file, il ripristino richiederà la riconfigurazione manuale di tutte le credenziali e la perdita dei dati produttivi.
