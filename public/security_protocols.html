1. Premessa e Ambito di Applicazione

1.1 Finalità del documento

Il presente documento definisce i protocolli di sicurezza informatica implementati nel Sistema MES (Manufacturing Execution System) in uso presso Grafica Nappa S.R.L. Il documento ha carattere ufficiale e costituisce riferimento per l'archiviazione aziendale delle misure di sicurezza adottate a protezione dei dati, delle infrastrutture e dei processi produttivi gestiti dal sistema.

1.2 Perimetro del sistema

Il perimetro di applicazione comprende:

  • Applicazione web MES (backend Laravel/PHP, frontend Blade/Bootstrap/jQuery)
  • Database MySQL su server dedicato (192.168.1.60)
  • Integrazioni con sistemi esterni: Onda IQ (ERP), Prinect (stampa offset), Fiery V900 (stampa digitale), BRT (spedizioni), NetTime (presenze)
  • Rete locale aziendale (non esposta a Internet)
  • Dispositivi client: PC operatore, tablet reparto, TV kiosk, smartphone

1.3 Riferimenti normativi

  • ISO/IEC 27001:2022 — Sistema di gestione della sicurezza delle informazioni
  • OWASP Top 10 (2021) — Principali vulnerabilità applicazioni web
  • Regolamento UE 2016/679 (GDPR) — Art. 32: Sicurezza del trattamento
  • OWASP Application Security Verification Standard (ASVS) v4.0

1.4 Destinatari

Direzione Aziendale, Amministrazione, consulenti esterni autorizzati.

3. Protocolli di Sicurezza Attivi

SEC-AUTH-001 Autenticazione multi-livello OPERATIVO
Sistema di autenticazione differenziato per ruolo. Gli operatori accedono tramite codice operatore univoco. Gli amministratori richiedono codice operatore e password con hash Bcrypt (12 round). L'autenticazione a due fattori (TOTP) è disponibile per il ruolo admin.
Componenti: OperatoreLoginController, AdminLoginController, Guard operatore — Rif. ISO 27001 A.9.4.2, OWASP A07:2021
SEC-AUTH-002 Token per-tab con scadenza OPERATIVO
Ad ogni login viene generato un token crittograficamente sicuro di 64 caratteri random, memorizzato nel database con scadenza a 12 ore. Il token isola le sessioni per-tab e viene invalidato al logout. La validità è verificata ad ogni richiesta tramite scope valido().
Componenti: OperatoreToken model, tabella operatore_tokens, indice su expires_at — Rif. OWASP A07:2021
SEC-AUTH-003 Autenticazione a due fattori (2FA TOTP) OPERATIVO
Autenticazione a due fattori basata su algoritmo TOTP (Time-based One-Time Password, RFC 6238) per utenti con ruolo admin. Il segreto 2FA è cifrato a riposo con AES-256-GCM. Implementa il meccanismo "dispositivo fidato": dopo la prima verifica, il dispositivo viene registrato con token SHA-256 e cookie persistente (10 anni). Sono disponibili 8 codici di recupero monouso.
Componenti: TwoFactorService, TwoFactorController, tabella trusted_devices, pacchetto pragmarx/google2fa — Rif. ISO 27001 A.9.4.2, NIST SP 800-63B
SEC-RBAC-001 Controllo accessi basato su ruoli (RBAC) OPERATIVO
Sistema RBAC con 9 ruoli e 23 permessi granulari implementato tramite spatie/laravel-permission v7.2. I ruoli sono assegnati a livello di modello Operatore con guard dedicato. L'enforcement avviene tramite 4 middleware personalizzati sulle route. La matrice ruoli-permessi è definita nel seeder e verificabile in ogni momento.
Componenti: HasRoles trait, RbacSeeder, middleware operatore.auth, owner, admin, owner.or.admin, 5 tabelle permission — Rif. ISO 27001 A.9.1.2
SEC-AUDIT-001 Audit trail centralizzato OPERATIVO
Sistema di tracciamento automatico degli eventi di sicurezza. Registra: login, logout, modifiche stato fasi di produzione, eliminazione record. Ogni entry include identificativo utente, azione, valori precedenti e successivi, indirizzo IP e user-agent. Retention: 90 giorni con pulizia automatica schedulata alle 03:00. Implementa graceful degradation: il fallimento del log non blocca l'operatività.
Componenti: AuditService, OrdineFaseObserver, tabella audit_logs (5 indici), comando audit:pulisci — Rif. ISO 27001 A.12.4.1, GDPR Art. 5(2)
SEC-RATE-001 Limitazione frequenza richieste (Rate Limiting) OPERATIVO
Protezione anti brute-force sugli endpoint di autenticazione. Login: massimo 5 tentativi in 15 minuti per indirizzo IP. API generiche: massimo 60 richieste al minuto per IP. Al superamento della soglia, il sistema risponde con codice HTTP 429 (Too Many Requests).
Componenti: RateLimiter in AppServiceProvider, middleware throttle:login su route POST login — Rif. OWASP A07:2021
SEC-HDR-001 Security Headers HTTP OPERATIVO
Middleware globale che aggiunge header di sicurezza HTTP a tutte le risposte dell'applicazione: X-Frame-Options (protezione clickjacking), X-Content-Type-Options (prevenzione MIME sniffing), Referrer-Policy (controllo informazioni referrer), Permissions-Policy (disabilitazione accesso camera, microfono, geolocalizzazione).
Componenti: SecurityHeaders middleware, registrato in bootstrap/app.php — Rif. OWASP A05:2021
SEC-CSRF-001 Protezione Cross-Site Request Forgery (CSRF) OPERATIVO
Protezione CSRF nativa Laravel con token sincronizzato. Il token viene rigenerato automaticamente ogni 30 minuti tramite endpoint dedicato (/csrf-refresh). Implementa intercettore fetch per auto-refresh e gestione navigazione browser (pageshow event). 6 endpoint esenti da CSRF utilizzano autenticazione alternativa tramite token operatore (op_token).
Componenti: middleware ValidateCsrfToken, endpoint /csrf-refresh, fetch interceptor JavaScript — Rif. OWASP A01:2021
SEC-SESS-001 Gestione sessioni sicura OPERATIVO
Sessioni archiviate in database (non su filesystem). Cifratura sessioni abilitata (AES-256-GCM via APP_KEY). Cookie di sessione con flag HttpOnly (inaccessibile a JavaScript) e SameSite=Lax (protezione CSRF cross-site). Timeout sessione: 480 minuti (8 ore lavorative).
Componenti: config/session.php, driver database, tabella sessions, SESSION_ENCRYPT=true — Rif. ISO 27001 A.14.1.2, OWASP A07:2021
SEC-DATA-001 Isolamento credenziali e segreti OPERATIVO
Tutte le credenziali di accesso ai sistemi esterni (Onda, Prinect, Fiery, BRT, NetTime, SMTP) sono archiviate nel file .env, escluso dal repository Git tramite .gitignore. La chiave di cifratura applicativa (APP_KEY) utilizza AES-256. Le password utente sono hashate con Bcrypt a 12 round. Il segreto 2FA è cifrato a riposo con encrypt().
Componenti: .env, .gitignore, Hash::make() Bcrypt, encrypt() AES-256-GCM — Rif. ISO 27001 A.10.1.1, OWASP A02:2021
SEC-VAL-001 Validazione e sanitizzazione input OPERATIVO
Validazione server-side su tutti gli endpoint che accettano dati utente tramite $request->validate(). Regole includono: vincoli di tipo (string, numeric, date), vincoli referenziali (exists:tabella), lunghezza massima, valori ammessi (in:lista). L'output è protetto tramite escape automatico Blade ({{ }}). Le istanze di output non-escaped ({!! !!}) sono limitate a 10 occorrenze verificate sicure (entità HTML statiche, json_encode(), funzione e()).
Componenti: validazione Laravel su tutti i controller, Blade auto-escaping — Rif. OWASP A03:2021

4. Controllo Accessi e Autenticazione

4.1 Flusso di autenticazione operatore

  1. L'operatore inserisce il proprio codice operatore univoco
  2. Il sistema verifica l'esistenza e lo stato attivo dell'operatore
  3. Viene creata la sessione con guard operatore
  4. Viene generato un token univoco di 64 caratteri (scadenza 12h)
  5. L'operatore viene reindirizzato alla dashboard appropriata in base al ruolo

4.2 Flusso di autenticazione admin

  1. L'admin inserisce codice operatore e password
  2. Il sistema verifica le credenziali (Bcrypt 12 round) e il ruolo admin
  3. Viene creata la sessione
  4. Se il 2FA è abilitato:
    • Se il dispositivo è già fidato (cookie SHA-256): accesso diretto
    • Se il dispositivo è nuovo: richiesta codice TOTP a 6 cifre
    • Dopo verifica: il dispositivo viene registrato come fidato (persistenza illimitata)
  5. Redirect alla dashboard admin

6.3 Politica di conservazione

Retention: 90 giorni. Pulizia automatica eseguita quotidianamente alle ore 03:00 tramite comando schedulato audit:pulisci. I record oltre la soglia di retention vengono eliminati definitivamente.

6.4 Consultazione

L'audit log è consultabile tramite le pagine /owner/audit-log e /admin/audit-log con filtri per azione, utente e data. Paginazione a 50 record per pagina.

7.2 Protezione CSRF

  • Token CSRF sincronizzato su tutte le form POST (Laravel VerifyCsrfToken)
  • Refresh automatico ogni 30 minuti tramite endpoint /csrf-refresh
  • Intercettore JavaScript per auto-refresh e gestione errore 419
  • Gestione navigazione back/forward browser (evento pageshow)
  • 6 endpoint esenti con autenticazione alternativa (op_token)

8.2 Isolamento credenziali

Tutte le credenziali di accesso ai sistemi esterni sono archiviate esclusivamente nel file .env sul server di produzione. Il file è escluso dal repository Git. Le credenziali non compaiono nel codice sorgente.

8.3 Output encoding

Il framework Blade applica escape automatico (htmlspecialchars) a tutto l'output tramite sintassi {{ }}. Le 10 istanze di output non-escaped ({!! !!}) sono state verificate e risultano sicure (entità HTML statiche, json_encode(), funzione e()).

9.2 Architettura di rete

Il sistema MES opera esclusivamente all'interno della rete locale aziendale (LAN). Non è esposto a Internet. L'accesso dall'esterno è possibile solo tramite tunnel VPN autorizzato (tnnl.in) con configurazione proxy trust.
