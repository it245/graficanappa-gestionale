2. Descrizione del Sistema

Il Sistema MES (Manufacturing Execution System) di Grafica Nappa srl è un'applicazione web progettata per la gestione completa del ciclo produttivo di una tipografia industriale. Il sistema copre tutte le fasi dalla ricezione dell'ordine alla spedizione del prodotto finito, integrando dati da molteplici sorgenti esterne (ERP Onda, Prinect Heidelberg, Fiery EFI, BRT) in un'unica piattaforma.

Obiettivi della v1.x (raggiunti)

  • Tracciabilità completa — Monitoraggio in tempo reale di ogni fase produttiva, dall'ordine alla spedizione
  • Integrazione ERP — Sincronizzazione bidirezionale con il gestionale Onda tramite SOAP
  • Ottimizzazione macchine — Dati di produzione da Prinect (offset) e Fiery (digitale) in tempo reale
  • Gestione spedizioni — Tracking BRT, gestione consegne e note bidirezionali
  • Report ore — Analisi ore lavorate vs previste con dati Prinect e fallback pivot operatore

Nuovi obiettivi v2.0

  • Schedulazione intelligente (Mossa 37) — Motore a simulazione eventi discreti per ottimizzazione setup e throughput su 11 macchine, con risparmio stimato di 85-93 ore/mese
  • Chat MES in tempo reale — Sistema di messaggistica interna con @menzioni per reparto e operatore, canali dedicati, via Laravel Reverb WebSocket
  • DDT PDF automatici — Generazione automatica documenti di trasporto su carta intestata pre-stampata, con riferimento ordine Maxtris da Google Drive
  • Interfaccia Enterprise — Design system completo con sidebar, topbar, dark mode, KPI cards, CSS variables per futura multi-tenancy SaaS
  • PWA installabile — Progressive Web App con service worker, pagina offline, push notifications browser e notifiche sonore AudioContext
  • Sicurezza avanzata — RBAC Spatie (9 ruoli, 23 permessi), 2FA TOTP per admin, audit log, security headers, rate limiting, session encryption AES-256-GCM
  • Monitoraggio presenze — Integrazione NetTime per timbrature badge e calcolo ore presenza
  • Kiosk TV — Visualizzazione su monitor in reparto con 3 pagine rotanti

Auto-terminazione intelligente: Il sync Prinect implementa una logica a 3 livelli:

Il flag terminata_manualmente impedisce qualsiasi sovrascrittura. Workstep COMPLETED senza actualStartDate terminano correttamente.

5.9 Laravel Reverb WebSocket

ParametroValore
TecnologiaLaravel Reverb (WebSocket nativo Laravel)
UtilizzoChat MES real-time, notifiche dashboard, aggiornamenti stati
CanaliPrivati per reparto, presence per chat, pubblici per notifiche globali

Architettura WebSocket: Laravel Reverb gira come processo daemon separato. Il client Echo (incluso nel layout mes.blade.php) si connette automaticamente. Se la connessione WebSocket non è disponibile, tutte le funzionalità real-time (chat, notifiche, aggiornamenti dashboard) passano automaticamente al fallback polling HTTP ogni 15 secondi.

7.10 Priorità Automatiche

Formula Priorità Base (v1.x)

La priorità base considera la data di consegna, il numero di fasi rimanenti e il flag priorita_manuale. Le fasi con priorita_manuale = true vengono sempre mostrate per prime.

Priorità Mossa 37 (v2.0)

Il PriorityService calcola una priorità composita a 4 livelli:

  1. Disponibilità: la fase è disponibile solo quando tutte le fasi precedenti nella sequenza produttiva sono completate (disponibile_m37 = true)
  2. Urgenza reale: giorni lavorativi residui prima della scadenza, classificati in fasce (critica, alta, media, bassa). L'urgenza è sempre prioritaria rispetto al batching
  3. Batch affinity: raggruppamento lavori simili per minimizzare i tempi di setup (stessa carta, stesso lavaggio, stessa configurazione macchina), con finestra ±5 giorni urgenza
  4. Sequenza ciclo: ordine naturale nel ciclo produttivo (10→999) per evitare starvation delle fasi a valle
Config files: La mappatura delle priorità è definita in config/fasi_priorita.php, le ore stimate per fase in config/fasi_ore.php, e la sequenza produttiva in config/sequenza_fasi.php.

7.12 Scheduler Mossa 37

Novità v2.0: Il motore di schedulazione "Mossa 37" è il cuore della v2.0. Utilizza una simulazione a eventi discreti per ottimizzare la sequenza produttiva su 11 macchine, minimizzando i tempi di setup e massimizzando il throughput. Il nome "Mossa 37" richiama la mossa vincente dell'AlphaGo: una decisione controintuitiva ma ottimale.

Architettura

  • SchedulerService: simulazione a eventi discreti con propagazione tra fasi successive
  • PriorityService: calcolo priorità composita a 4 livelli (disponibilità, urgenza reale, batch affinity, sequenza ciclo)
  • SchedulerExportService: export del piano schedulato in formato Excel per condivisione
  • PhaseCompleted Event: trigger automatico al completamento fase per ricalcolo disponibilità delle fasi successive
  • Propagazione: il completamento di una fase rende automaticamente disponibili (disponibile_m37 = true) le fasi successive nella sequenza produttiva
  • disponibile_da: calcolo del timestamp a partire dal quale una fase può essere schedulata (basato sul completamento delle precedenti)
  • Date BRT: predizione date di spedizione basate sullo scheduling delle fasi finali

11. Analisi dei Rischi

RischioProbabilitàImpattoMitigazione
Interruzione sync Onda (SOAP) Media Alto Retry automatico, log errori, notifica admin via audit log, dati persistenti anche offline. Il MES continua a funzionare con dati già sincronizzati.
Downtime server .60 Bassa Critico Backup MySQL giornaliero (mysqldump), procedura di ripristino documentata (sez. 12), InnoDB crash recovery nativo.
Perdita connessione Prinect/Fiery Media Medio Fallback su dati manuali, ultima sync valida mantenuta, auto-terminazione intelligente protegge da stati inconsistenti. Flag terminata_manualmente per override.
Accesso non autorizzato Bassa Alto 2FA TOTP per admin, RBAC 9 ruoli / 23 permessi, rate limiting 5/15min, audit log completo, session encryption AES-256-GCM, security headers.
Crash MySQL / corruzione dati Bassa Critico InnoDB con crash recovery automatico, backup giornaliero, replica possibile, sync Onda/Prinect/Fiery ricostruiscono dati automaticamente.
WebSocket/Reverb non disponibile Media Basso Fallback automatico su polling HTTP (15s). Chat, notifiche e aggiornamenti dashboard funzionano anche senza WebSocket, con latenza leggermente superiore.
Conflitto dati Excel bidirezionale Media Medio Timestamp-based conflict resolution, MES come source of truth. Log delle sovrascritture per debug.
Overload scheduler Mossa 37 Bassa Medio Timeout configurabile, esecuzione asincrona via Laravel queue, log dettagliato con sched_calcolato_at timestamp.
Dipendenza Laravel Reverb Media Basso Reverb gira come processo daemon con auto-restart. Fallback polling garantisce funzionalità base. Chat e notifiche non sono critiche per la produzione.
Costi API Claude (v3.0) Media Basso Stimato ~$25-50/mese con Sonnet (~40 chiamate/giorno), monitoraggio costi, budget cap configurabile.
Dipendenza Google Drive Bassa Basso Solo per RIF. ORD. MAXTRIS nei DDT; DDT generabile anche senza il riferimento. Copia locale come cache.
Interruzione NetTime / share SMB Media Basso Le presenze non sono critiche per la produzione. Il MES funziona normalmente senza dati timbrature.
Risolto in v2.0: Il rischio "Database SQLite — concorrenza limitata e locking" presente nelle versioni 1.x è stato completamente risolto con la migrazione a MySQL 8.x, che supporta transazioni ACID, locking a livello di riga e connessioni concorrenti illimitate.

12. Procedure di Ripristino

12.1 Backup MySQL

# Backup completo database (singola transazione per consistenza)
mysqldump -u root -p mes_production --single-transaction --routines --triggers > backup_mes_$(date +%Y%m%d).sql

# Backup automatico (cron giornaliero alle 02:00)
0 2 * * * mysqldump -u root -p*** mes_production --single-transaction --routines > /backup/mes_$(date +\%Y\%m\%d).sql

# Backup con compressione (per risparmiare spazio)
mysqldump -u root -p mes_production --single-transaction | gzip > /backup/mes_$(date +%Y%m%d).sql.gz

# Retention: mantenere almeno 30 giorni di backup

12.2 Ripristino da Backup MySQL

# 1. Mettere il sito in manutenzione
php artisan down

# 2. Fermare Reverb (WebSocket)
# Terminare il processo reverb:start

# 3. Ripristinare il database
mysql -u root -p mes_production < backup_mes_20260408.sql

# 4. Eseguire eventuali migrazioni mancanti
php artisan migrate --force

# 5. Pulire cache
php artisan config:cache
php artisan route:cache
php artisan view:cache

# 6. Riavviare i servizi
php artisan up
php artisan reverb:start --daemon

# 7. Verificare lo scheduler
php artisan schedule:run

12.3 Ripristino Codice da Git

# Ripristino dal branch def2.0
git fetch origin
git checkout def2.0
git reset --hard origin/def2.0

# Reinstallare dipendenze
composer install --no-dev --optimize-autoloader

# Eseguire migrazioni e ricostruire cache
php artisan migrate --force
php artisan config:cache
php artisan route:cache
php artisan view:cache

# Riavviare Reverb
php artisan reverb:start --daemon
Attenzione: In caso di corruzione grave del database, contattare immediatamente il Responsabile IT (Giovanni Pietropaolo). Il backup giornaliero garantisce una perdita massima di 24 ore di dati. Le sync automatiche Onda, Prinect e Fiery ricostruiscono i dati di produzione in poche ore dopo il ripristino.
