2. Descrizione del Sistema

Il Sistema MES (Manufacturing Execution System) di Grafica Nappa srl e un'applicazione web progettata per la gestione completa del ciclo produttivo di una tipografia industriale. Il sistema copre tutte le fasi dalla ricezione dell'ordine alla spedizione del prodotto finito.

Obiettivi Principali

  • Tracciabilita completa — Monitoraggio in tempo reale di ogni fase produttiva, dall'ordine alla spedizione
  • Integrazione ERP — Sincronizzazione bidirezionale con il gestionale Onda tramite SOAP
  • Ottimizzazione macchine — Dati di produzione da Prinect (offset) e Fiery (digitale) in tempo reale
  • Gestione spedizioni — Tracking BRT, generazione DDT, gestione consegne
  • Schedulazione intelligente — Motore Mossa 37: simulazione a eventi discreti per ottimizzazione setup e throughput
  • Comunicazione interna — Chat MES in tempo reale con menzioni per reparto e operatore
  • Generazione documenti — DDT PDF automatici su carta intestata pre-stampata
  • Interfaccia enterprise — Design system coerente con sidebar, dark mode, KPI cards
  • Accessibilita mobile — PWA installabile con notifiche push e modalita offline
  • Monitoraggio presenze — Integrazione NetTime per timbrature e ore lavorate

7.1 Dashboard Operatore

Interfaccia principale per gli operatori di reparto. Mostra le fasi assegnate ordinate per priorita, con possibilita di avviare, mettere in pausa e completare le lavorazioni.

7.4 Ciclo di Vita Fase

Ogni fase di lavorazione attraversa gli stati seguenti:

StatoCodiceDescrizioneColore Badge
Non Iniziata0Fase registrata, non ancora avviabileGrigio
Pronto1Fase pronta per essere avviata (materiali disponibili)Giallo
Avviato2Fase in lavorazione da un operatoreBlu
Terminato3Fase completataVerde
Consegnato4Materiale consegnato al reparto successivo o al clienteVerde scuro
Esterno5Fase inviata a fornitore esternoViola
Flag terminata_manualmente: Quando un operatore termina manualmente una fase, il flag terminata_manualmente viene impostato a true. Questo impedisce che la sync Prinect sovrascriva lo stato della fase.
Ricalcolo commessa: La funzione ricalcolaCommessa() ricalcola gli stati su tutti gli ordini della commessa utilizzando config/fasi_priorita.php. Viene invocata dopo ogni cambio stato fase.

7.12 Scheduler Mossa 37

Novita v2.0: Il motore di schedulazione "Mossa 37" e il cuore della v2.0. Utilizza una simulazione a eventi discreti per ottimizzare la sequenza produttiva su 11 macchine, minimizzando i tempi di setup e massimizzando il throughput.

Architettura

  • PriorityService: calcolo priorita a 4 livelli (disponibilita, urgenza reale, batch affinity, sequenza ciclo)
  • SchedulerService: simulazione eventi discreti con propagazione tra fasi
  • PhaseCompleted Event: trigger automatico al completamento fase per ricalcolo disponibilita successive
  • Propagazione: completamento di una fase rende disponibili le successive nella sequenza

Sequenza Produttiva

10 Stampa→ 11 Digitale→ 20 Plastica→ 30 Caldo JOH→ 31 Plast. Lux→ 35 Finit. Dig.→ 37 Taglio→ 39 Rilievi BOBST→ 40 Fustella→ 100 Finestre→ 110 Piegaincolla→ 120 Legatoria→ 700 Allestimento→ 999 Spedizione

Batching Intelligente

MacchinaStrategia BatchFinestra
BOBSTBlocchi consecutivi (2 config: rilievi seq39 + fustelle seq40), cambio 1h±5gg urgenza
PiegaincollaBlocchi consecutivi (3 config: PI01/PI02/PI03), cambio 1h±5gg urgenza
XL106Per tipo lavaggio + codice carta±5gg urgenza
PlastificatriciPer formato foglio±5gg urgenza

Colli di Bottiglia

In ordine di criticita: JOH (stampa a caldo) > BOBST (fustella/rilievi) > Piegaincolla. Raccomandazione: 3° turno per JOH.

Performance

  • Risparmio setup stimato: 85-93 ore/mese
  • Export Excel del piano schedulato
  • Turni: XL106 24h lun-ven, altre macchine 6-22 lun-ven
  • Predizione date BRT per spedizioni

8. Procedura di Deploy

Prerequisiti

  • PHP 8.5 con estensioni: pdo_mysql, soap, gd, mbstring, openssl, zip
  • MySQL 8.x attivo sul server 192.168.1.60
  • Composer 2.x installato
  • Git configurato
  • Accesso rete a Onda (SOAP), Prinect (REST), Fiery (REST), NetTime (SMB)

Procedura

# 1. Clonare/aggiornare il repository
git checkout def2.0
git pull origin def2.0

# 2. Installare dipendenze (include dompdf, pusher, reverb, spatie)
composer install --no-dev --optimize-autoloader

# 3. Configurare ambiente
cp .env.example .env
# Editare .env con credenziali DB, Onda, Prinect, Fiery, NetTime, Reverb

# 4. Eseguire migrazioni
php artisan migrate --force

# 5. Cache configurazione
php artisan config:cache
php artisan route:cache
php artisan view:cache

# 6. Avviare lo scheduler (cron o Task Scheduler Windows)
php artisan schedule:run

# 7. Avviare Laravel Reverb (WebSocket)
php artisan reverb:start --daemon

# 8. Seed permessi e ruoli (prima installazione)
php artisan db:seed --class=RolesAndPermissionsSeeder
Importante: Tutti i comandi artisan (migrate, sync, tinker) vanno eseguiti sul server .60, mai in locale. Il database MySQL risiede esclusivamente sul server di produzione.

11. Analisi dei Rischi

RischioProbabilitaImpattoMitigazione
Interruzione sync Onda (SOAP) Media Alto Retry automatico, log errori, notifica admin, dati persistenti anche offline
Downtime server .60 Bassa Critico Backup MySQL giornaliero, procedura di ripristino documentata (sez. 12)
Perdita connessione Prinect/Fiery Media Medio Fallback su dati manuali, ultima sync valida mantenuta, auto-terminazione intelligente
Accesso non autorizzato Bassa Alto 2FA TOTP admin, RBAC 9 ruoli, rate limiting, audit log, session encryption AES-256-GCM
Crash MySQL / corruzione dati Bassa Critico InnoDB con crash recovery, backup giornaliero, replica possibile
WebSocket/Reverb non disponibile Media Basso Fallback automatico su polling HTTP (15s). Chat e notifiche funzionano anche senza WebSocket
Conflitto dati Excel bidirezionale Media Medio Timestamp-based conflict resolution, MES come source of truth
Overload scheduler M37 Bassa Medio Timeout configurabile, esecuzione asincrona via queue, log dettagliato
Costi API Claude (v3.0) Media Basso Stimato ~$25-50/mese con Sonnet (~40 chiamate/giorno), monitoraggio costi
Dipendenza Google Drive Bassa Basso Solo per RIF. ORD. MAXTRIS; DDT generabile anche senza
Risolto in v2.0: Il rischio "Database SQLite — concorrenza limitata e locking" presente nelle versioni 1.x e stato completamente risolto con la migrazione a MySQL 8.x, che supporta transazioni ACID, locking a livello di riga e connessioni concorrenti illimitate.

12. Procedure di Ripristino

12.1 Backup MySQL

# Backup completo database
mysqldump -u root -p mes_production --single-transaction --routines > backup_mes_$(date +%Y%m%d).sql

# Backup automatico (cron giornaliero alle 02:00)
0 2 * * * mysqldump -u root -p*** mes_production --single-transaction > /backup/mes_$(date +\%Y\%m\%d).sql

12.2 Ripristino da Backup

# 1. Fermare i servizi
php artisan down

# 2. Ripristinare il database
mysql -u root -p mes_production < backup_mes_20260408.sql

# 3. Eseguire eventuali migrazioni mancanti
php artisan migrate --force

# 4. Pulire cache
php artisan config:cache
php artisan route:cache
php artisan view:cache

# 5. Riavviare i servizi
php artisan up
php artisan reverb:start --daemon
php artisan schedule:run

12.3 Ripristino Codice

# Ripristino da Git
git fetch origin
git checkout def2.0
git reset --hard origin/def2.0
composer install --no-dev --optimize-autoloader
php artisan migrate --force
php artisan config:cache
Attenzione: In caso di corruzione grave del database, contattare immediatamente il Responsabile IT. Il backup giornaliero garantisce una perdita massima di 24 ore di dati. Le sync Onda, Prinect e Fiery ricostruiscono i dati di produzione automaticamente.
