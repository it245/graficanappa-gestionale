1. Concetti di base — Rete e IP

Prima di parlare di tunnel, capiamo perché servono. Ogni macchina su una rete ha un indirizzo IP. Esistono due tipi:

  • IP privato (es. 192.168.1.60): visibile solo dentro la stessa rete locale (LAN). Il server Grafica Nappa è 192.168.1.60 e lo raggiungi solo se sei in azienda.
  • IP pubblico (es. 93.47.8.102): assegnato dal provider internet al router. Visibile da tutto il mondo.

Il problema: Prinect, Fiery, server MES hanno IP privati. Da casa tua non puoi chiamarli direttamente. Serve un ponte che dalla tua rete ti porti dentro la rete aziendale. Questo ponte si chiama VPN o tunnel.

Casa tua (82.xx.xx.xx) Azienda (93.47.8.102) │ │ │ INTERNET │ │ ─────────────────── │ │ │ │ │ │ │ ┌───────────┐ │ │ │ └──┤ VPN / TUN │──┘ │ │ └───────────┘ │ │ │ PC tuo ═══════════════════> Server 192.168.1.60 Prinect 192.168.1.10 Fiery 192.168.1.20

2. Scenari tipici — Dove si trova il MES

2.1 Scenario A — MES dentro la rete cliente (on-premise)

È la situazione attuale di Grafica Nappa. Il server MES è fisicamente in azienda, stessa rete di Prinect/Fiery. Le chiamate API sono locali e istantanee.

  • ✓ Semplice, veloce, basso costo
  • ✓ Zero problemi di firewall
  • ✓ Da casa raggiungi il server tramite VPN WireGuard integrata nel router MikroTik aziendale (setup già attivo: tunnel "NappaG", endpoint *.mynetname.net, IP VPN assegnato 192.168.216.x)
  • Alternativa secondaria: RDP al tuo PC ufficio e lavori da lì
Nota pratica — setup attuale Grafica Nappa. Il router MikroTik in ufficio ha WireGuard server attivo su porta 27233. Ogni sviluppatore/operatore autorizzato ha un peer dedicato con chiavi proprie. Dal client WireGuard sul PC di casa, Attiva il tunnel "NappaG" → ricevi IP 192.168.216.x → puoi raggiungere 192.168.1.60 (server MES) e tutta la LAN aziendale. Questa è la stessa architettura che proporremo ai clienti.

2.2 Scenario B — MES in cloud (VPS Hetzner/Aruba)

Il server MES è su un provider cloud, accessibile via dominio mes.cliente.com. Le macchine del cliente restano in azienda.

  • ✓ Tu lavori senza VPN: apri browser, entri
  • ✓ Backup e manutenzione gestiti dal provider
  • ✗ Il MES cloud deve raggiungere Prinect/Fiery dentro la rete cliente → serve tunnel tra cloud e cliente

2.3 Scenario C — Ibrido

Database e dashboard in cloud, un piccolo "agente" installato nella rete cliente che fa da ponte per le API macchine.

4. Tailscale — La soluzione consigliata per iniziare

Tailscale è una mesh VPN costruita su WireGuard. Si installa in 5 minuti, funziona su qualsiasi dispositivo, è gratis fino a 100 dispositivi. Tutti i nodi della tua "tailnet" si vedono come fossero nella stessa LAN, anche se sono sparsi per il mondo.

Perché Tailscale. Niente port-forwarding, niente firewall da configurare, niente IP pubblico statico. Ogni dispositivo ottiene un IP 100.x.x.x privato della rete Tailscale, cifrato end-to-end. Scali da 1 a 100 dispositivi senza cambiare nulla.

4.1 Setup Tailscale per Grafica Nappa

Crea account Tailscale

Vai su https://tailscale.com e registrati con Google/GitHub/Microsoft. Account personal gratis include 100 dispositivi, 3 utenti. Per crescere: piano team 6$/mese.

Installa Tailscale sul server Grafica Nappa (.60)

# Su Windows Server .60:
# Scarica installer da https://tailscale.com/download/windows
# Esegui come Admin, login con il tuo account
# Avvia servizio Tailscale (si avvia da solo)

# Verifica IP assegnato:
tailscale ip -4
# Esempio output: 100.112.34.56

Installa Tailscale sul tuo PC di casa

# Windows/Mac/Linux: scarica client dal sito
# Login con STESSO account Tailscale
# Verifica nodi connessi:
tailscale status

# Ora puoi pingare il server come se fosse in LAN:
ping 100.112.34.56
# Oppure dal nome:
ping graficanappa-server

Apri il MES dal tuo PC di casa

# Nel browser:
http://100.112.34.56/
# Oppure:
http://graficanappa-server/

# Tutto come se fossi in azienda, senza aprire porte.

Abilita subnet routing (per raggiungere Prinect/Fiery)

Tailscale di default connette solo i dispositivi con l'agent installato. Ma Prinect e Fiery non possono installare Tailscale. Soluzione: il server .60 diventa "router" per gli altri IP della sua LAN.

# Sul server .60 (PowerShell come Admin):
tailscale up --advertise-routes=192.168.1.0/24

# Poi vai su tailscale.com/admin, vedi subnet in attesa
# di approvazione → clicca "Approve"

# Ora dal tuo PC di casa:
ping 192.168.1.10   # Prinect funziona
ping 192.168.1.20   # Fiery funziona

Abilita MagicDNS (nomi al posto di IP)

Nel pannello Tailscale admin attiva "MagicDNS". D'ora in poi pinga per nome: ping server-60.

Nota sicurezza. Chi ha l'account Tailscale del server vede tutta la rete aziendale. Usa 2FA obbligatorio, chiavi di autenticazione a scadenza, ACL per limitare chi vede cosa. Mai condividere credenziali.

5. VPN classica — WireGuard (alternativa self-hosted)

Se non vuoi dipendere da un servizio esterno, WireGuard è la VPN moderna self-hosted. Codice minuscolo (~4000 righe), performance native del kernel Linux, cifratura moderna.

5.1 Quando scegliere WireGuard invece di Tailscale

  • Vuoi controllo totale (niente server di terzi)
  • Il cliente richiede VPN interna aziendale (es. per compliance)
  • Hai già un server pubblico e sai configurarlo

5.2 Setup essenziale

# Sul server pubblico (es. VPS 20€/mese):
apt install wireguard

# Genera chiavi:
wg genkey | tee privatekey | wg pubkey > publickey

# Configurazione /etc/wireguard/wg0.conf:
[Interface]
Address = 10.10.0.1/24
ListenPort = 51820
PrivateKey = <server-private-key>

[Peer]
PublicKey = <client-public-key>
AllowedIPs = 10.10.0.2/32

# Avvia:
wg-quick up wg0
systemctl enable wg-quick@wg0

Lato client simile, ma con IP cliente e PublicKey del server. La curva di apprendimento è maggiore di Tailscale, ma nessuna dipendenza esterna.

6. VPN site-to-site — Per produzione seria

Scenario: MES in cloud (Hetzner), macchine nel cliente (Grafica Nappa). Serve tunnel permanente tra i due siti, trasparente per le applicazioni.

┌─────────────────────┐ ┌─────────────────────┐ │ VPS Cloud MES │ │ Router Cliente │ │ IP pubblico │◄────────►│ (pfSense/MikroTik) │ │ WireGuard server │ TUNNEL │ WireGuard client │ │ │ │ │ │ MES app │ │ LAN 192.168.1.x │ │ legge Prinect via │ │ ├─ Prinect .10 │ │ 192.168.1.10 come │ │ ├─ Fiery .20 │ │ se fosse locale │ │ └─ Server .60 │ └─────────────────────┘ └─────────────────────┘

Come funziona

  1. Router cliente ha VPN integrata (quasi tutti quelli seri: pfSense, OPNsense, MikroTik, FortiGate, Ubiquiti Unifi Security Gateway)
  2. VPS cloud configurato come "peer" WireGuard
  3. Router cliente inizia la connessione (outbound) → nessuna porta aperta sul firewall cliente
  4. Una volta stabilita, il VPS può chiamare qualsiasi IP della rete cliente
  5. L'MES in cloud fa curl http://192.168.1.10/api/status come se fosse in loco

Pro

  • Connessione stabile 24/7, riconnessione automatica
  • Zero configurazione sui dispositivi end (Prinect/Fiery non sanno niente)
  • Gestione centralizzata via router

Contro

  • Serve router cliente che supporti VPN (il router domestico Fastweb/TIM spesso NO)
  • Setup iniziale richiede competenze di networking
  • Se cambia l'IP pubblico del cliente, tunnel si disconnette (usare DynDNS)

7. Reverse tunnel — Quando il cliente non ha router moderno

A volte il cliente ha solo un modem di base, zero controllo sul firewall. Soluzione: reverse tunnel. Un piccolo software dentro la rete cliente apre una connessione uscente verso il cloud. Nessuna porta in ingresso, nessun firewall da toccare.

7.1 Cloudflare Tunnel (raccomandato)

Gratis, robusto, integrato con Cloudflare DNS/SSL. Funziona così:

Crea account Cloudflare + aggiungi dominio

Serve un dominio (anche 5€/anno su Namecheap). Lo punti a Cloudflare cambiando i nameserver.

Installa cloudflared sul server cliente

# Windows: scarica exe da github.com/cloudflare/cloudflared
# Login:
cloudflared tunnel login

# Crea tunnel:
cloudflared tunnel create mes-cliente

# Configura routing:
cloudflared tunnel route dns mes-cliente mes.dominio.it

Avvia come servizio

cloudflared service install <token>
# Ora mes.dominio.it → server interno cliente
# Nessuna porta aperta, HTTPS automatico

7.2 ngrok (per demo/test)

Strumento veloce per esporre un servizio locale al web. Gratis con limiti (sessione 8h, URL random). Ottimo per far vedere al cliente una feature in 30 secondi senza deploy.

ngrok http 80
# Output:
# Forwarding https://abc123.ngrok.io -> http://localhost:80

8. Agente locale — Soluzione enterprise

Se vuoi architettura cloud "pura" senza VPN, installa un piccolo agente sul server del cliente. L'agente:

  • Si collega in uscita al MES cloud tramite WebSocket
  • Riceve comandi dal cloud ("chiamami Prinect", "leggi Fiery status")
  • Esegue in locale, restituisce la risposta
  • Nessuna porta aperta, traffico solo uscente, massima sicurezza

Costo: devi sviluppare tu l'agente (1-2 settimane di lavoro). Lo fai una volta, lo riusi su tutti i clienti.

9. Macchine che pushano dati (MQTT)

Le macchine industriali più recenti (BOBST serie iQ, HP Indigo, Konica nuove) supportano pubblicazione via MQTT. Invece che tu interroghi la macchina, la macchina manda i dati a un broker centrale.

BOBST iQ ──publish──► ┌──────────────┐ ◄──subscribe── MES Cloud HP Indigo ──publish──► │ MQTT Broker │ ◄──subscribe── Dashboard Konica AccurioPress──publish──►│ (cloud) │ ◄──subscribe── Mobile App └──────────────┘

Solo traffico uscente dal cliente, broker in cloud, ogni app si sottoscrive ai topic che gli interessano. Scala perfettamente. Tecnologie: Mosquitto (open source), HiveMQ, AWS IoT Core.

10. Flusso di lavoro quotidiano da remoto

Dopo aver configurato Tailscale, ecco come si svolge una tua giornata tipica:

10.1 Mattina — Lavoro su nuova feature

# Dal tuo PC di casa, apri terminale:
cd /percorso/graficanappa-gestionale
git pull origin master

# Avvia Tailscale (di solito in autostart)
# Test connessione al server:
ping 192.168.1.60

# Connetti MySQL Workbench al server .60 via IP privato
# Scrivi codice, testa su DB produzione (attenzione!)
# Oppure usa DB locale per test:
php artisan serve

# Quando fix è pronto:
git commit -am "Fix priorità manuale persa"
git push origin master

# Deploy sul server:
ssh giovanni@192.168.1.60
cd /path/to/mes
git pull
php artisan migrate --force
exit

10.2 Emergenza alle 22:00 — Operatore segnala bug

  1. Ti scrivono su WhatsApp: "MES non risponde"
  2. Da casa: ping 192.168.1.60 → funziona
  3. SSH/RDP al server → controlli log storage/logs/laravel.log
  4. Trovi errore → fix rapido → deploy → telefoni operatore "è a posto"

Senza Tailscale/VPN: dovresti andare fisicamente in azienda. Con Tailscale: 10 minuti da casa.

10.3 Installazione presso nuovo cliente

  1. Prima visita in loco: installi Tailscale sul loro server, testi da remoto
  2. Tornato in Nappa: via Tailscale installi MES, configuri, testi
  3. Per il cliente cloud: installi anche WireGuard sul router loro → MES cloud parla con Prinect loro
  4. Training operatori: in video call + accesso RDP al loro server per troubleshooting

11. Sicurezza — Regole che non puoi rompere

Mai fare queste cose. Ogni punto è una potenziale catastrofe.
  • Mai esporre il MES direttamente a internet senza HTTPS + autenticazione forte. Apri solo tramite VPN o reverse tunnel con auth.
  • Mai aprire porte di Prinect/Fiery a internet. Quelle API non hanno autenticazione enterprise, chiunque le raggiunge fa danni.
  • Mai condividere credenziali VPN via WhatsApp/email. Usa un password manager (Bitwarden, 1Password).
  • Mai commitare file .env, chiavi SSH, token VPN in git. Usa .gitignore.
  • Mai usare password banali su account VPN/Tailscale. Minimo 16 caratteri, gestore password + 2FA.
  • Mai lasciare sessioni RDP aperte quando chiudi il PC. Disconnetti esplicitamente.
  • Mai installare software sul server cliente senza autorizzazione scritta (email conta).

11.1 Regole positive da seguire

  • Attiva 2FA su tutti gli account (GitHub, Tailscale, Cloudflare, email)
  • Password manager obbligatorio, niente password123
  • Log accessi: controlla chi è entrato, da dove, quando
  • Key rotation annuale: cambia chiavi SSH/VPN una volta l'anno
  • Backup prima di ogni operazione distruttiva (migration, reset)
  • Ambiente di staging separato dalla produzione — mai testare su dati veri
  • Quando il cliente ti dà accesso, documenta (email) cosa farai e per quanto

Sul tuo PC

  • Git installato (git --version)
  • PHP 8.5 + Composer (per lavorare in locale)
  • MySQL / Laragon / XAMPP
  • VS Code o PhpStorm
  • Tailscale client
  • Client SSH (Windows Terminal / PuTTY)
  • RDP client (Remote Desktop Connection, nativo Windows)
  • MySQL Workbench o HeidiSQL
  • Password manager (Bitwarden, gratis)
  • Chiave SSH personale (ssh-keygen) caricata su GitHub

Sui server aziendali

  • Tailscale installato sul server .60
  • Tailscale con subnet routing attivo per 192.168.1.0/24
  • Utente SSH/RDP personale (no login condivisi)
  • Il tuo user GitHub con accesso al repo
  • Accesso MySQL via account dedicato (no root in produzione)

Sul cliente nuovo

  • Un dominio dedicato (es. mes.cliente.com) o IP pubblico
  • Server dedicato o VPS con Linux/Windows
  • Tailscale installato (dopo firma contratto)
  • Accesso VPN fornito in modo sicuro (Bitwarden share)
  • Contatto tecnico del cliente con permessi admin
  • Backup iniziale dei loro dati (se esistono) prima di importare in MES

13. Glossario minimo

API (Application Programming Interface)
Modo per far parlare due software. Prinect espone API REST → il MES chiama e ottiene dati.
LAN (Local Area Network)
Rete locale dell'azienda. Tutti i dispositivi con IP 192.168.x.x o 10.x.x.x.
NAT (Network Address Translation)
Meccanismo del router: traduce IP privati in IP pubblico quando esci su internet. Motivo per cui non raggiungi dall'esterno un IP 192.168.x.
Port forwarding
Aprire una porta del router per instradare richieste esterne verso un dispositivo interno. Pericoloso se fatto male. Meglio VPN.
VPN (Virtual Private Network)
Tunnel cifrato che collega il tuo dispositivo a una rete remota come se fossi lì fisicamente.
WireGuard
Protocollo VPN moderno, veloce, semplice. Base di Tailscale.
Mesh VPN
VPN dove ogni dispositivo si connette direttamente a ogni altro (peer-to-peer), non tramite un server centrale. Tailscale è mesh.
Reverse tunnel
Connessione aperta dall'interno verso l'esterno per esporre un servizio locale al web senza aprire porte.
SSH (Secure Shell)
Protocollo per connettersi a un server Linux via riga di comando in modo sicuro.
RDP (Remote Desktop Protocol)
Come SSH ma per Windows, con interfaccia grafica completa.
MQTT
Protocollo leggero per IoT. Dispositivi pubblicano messaggi a un broker, altri si sottoscrivono ai topic che gli interessano.
OPC-UA
Standard industriale per comunicare con macchine di produzione. Tipico in fustellatrici BOBST moderne.
JDF (Job Definition Format)
Formato XML standard nell'industria della stampa per descrivere un lavoro (quantità, colori, finitura). Gestionali generano JDF, macchine lo leggono.

15. Cosa fare adesso — Piano d'azione

  1. Oggi: crea account Tailscale, installa client sul tuo PC, installa sul server .60. Fai pingare da casa.
  2. Questa settimana: configura subnet routing per raggiungere Prinect (.10) e Fiery (.20) da casa. Testa chiamata API da browser esterno.
  3. Due settimane: installa Bitwarden personale, metti dentro tutte le password aziendali. 2FA ovunque.
  4. Un mese: leggi la documentazione Prinect API e Fiery API — capisci cosa il MES già usa e cosa si potrebbe usare in più.
  5. Prima del nuovo cliente: prepara installer script (install.sh) che setta tutto in un comando. Provalo su una VM di test.
Conclusione. Il lavoro remoto sicuro è questione di 3 elementi: un tunnel per entrare nella rete, credenziali sicure per autenticarti, regole di sicurezza per non fare danni. Tailscale ti dà il primo in 10 minuti. Bitwarden il secondo in 20. Le regole del capitolo 11 le memorizzi in una settimana.
